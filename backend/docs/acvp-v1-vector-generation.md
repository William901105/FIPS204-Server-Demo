# ACVP v1 Vector Generation

Phase 3-4 added deterministic local skeleton vector generation from negotiated ML-DSA capabilities. Phase 3-5 keeps that behavior and attaches it to the formal local testSession/vectorSet state machine. Phase 4-1 persists generated sessions, vector sets, submissions, results, reports, and state events in SQLite. Phase 5-1 adds an explicit `nist-conformance` generation profile for ML-DSA minimum-count and coverage testing. This is not a production-ready ACVP server.

References:

- ACVP Protocol Specification: https://pages.nist.gov/ACVP/draft-fussell-acvp-spec.html
- ACVP ML-DSA JSON Specification: https://pages.nist.gov/ACVP/draft-celi-acvp-ml-dsa.html
- ACVP documentation landing page: https://pages.nist.gov/ACVP/
- FIPS 204: https://csrc.nist.gov/pubs/fips/204/final

Every response still includes:

```json
{
  "productionReady": false,
  "profile": "local-fips204-skeleton",
  "demoOnly": true,
  "notProductionAcvp": true
}
```

## Deterministic Generation

The local generator derives vector data with `SHA256(seed_material || label || counter)` and emits uppercase hex. It does not use `os.urandom`, `random`, wall-clock time, or session/vector-set UUIDs for vector bytes.

`campaignSeed` controls reproducibility. It must be a 16 to 64 byte hex string when supplied. If omitted, the server uses:

```text
SHA256(canonical JSON registration + "FIPS204-ACVP-PHASE-3-4")
```

`generationProfile` defaults to `local-debug`. In `local-debug`, `testsPerGroup` defaults to `2` and is capped at `10` to keep local oracle work bounded.

In `nist-conformance`, `testsPerGroup` is not capped by the local debug limit. Low values are automatically raised to the NIST minimum for each mode: keyGen 25, sigGen 15, and sigVer 15. See `backend/docs/mldsa-conformance-vector-generation.md` for the Phase 5-1 conformance profile details.

This deterministic generator is a local skeleton mechanism for reproducible demos. It is not a production entropy source.

## Create And Generate

By default, registration requests now generate vector sets immediately:

```json
{
  "algorithms": [
    {
      "algorithm": "ML-DSA",
      "mode": "keyGen",
      "revision": "FIPS204",
      "parameterSets": ["ML-DSA-44"]
    }
  ],
  "campaignSeed": "00112233445566778899AABBCCDDEEFF",
  "testsPerGroup": 2,
  "generationProfile": "local-debug",
  "autoGenerateVectorSets": true
}
```

Response shape:

```json
{
  "status": "vectorReady",
  "vectorSetIds": ["uuid"],
  "vectorGeneration": {
    "campaignSeed": "00112233445566778899AABBCCDDEEFF",
    "testsPerGroup": 2,
    "effectiveMinimums": {
      "keyGen": 2,
      "sigGen": 2,
      "sigVer": 2
    },
    "generatedVectorSetCount": 1,
    "modes": ["keyGen"],
    "generationProfile": "local-debug",
    "localSkeletonBehavior": true
  }
}
```

To preserve Phase 3-3 behavior, set:

```json
{
  "autoGenerateVectorSets": false
}
```

The session remains `capabilitiesAccepted` with empty `vectorSetIds`.

## Explicit Generation

For a `capabilitiesAccepted` session, generate vectors later:

```text
POST /acvp/v1/testSessions/{sessionId}/vectorSets/generate
```

Request:

```json
{
  "campaignSeed": "00112233445566778899AABBCCDDEEFF",
  "testsPerGroup": 2,
  "generationProfile": "local-debug"
}
```

If vector sets already exist, the endpoint returns `409 VECTOR_SETS_ALREADY_GENERATED`. If the session is cancelled or expired, the endpoint returns a `409` local state-machine error. Prompt-based sessions return `409 NEGOTIATED_CAPABILITIES_NOT_AVAILABLE`.

Successful generation records `stateHistory` events:

- session `capabilitiesAccepted -> vectorReady`
- session `vectorGenerated`
- vector set `created -> ready`

Generated vector sets inherit the session `expiresAt` when no explicit generation expiration is supplied. `expiresInSeconds` on the explicit generate request sets the expiration for generated vector sets.

## Generated ML-DSA Prompts

keyGen:

- one test group per negotiated `parameterSet`
- each test includes a deterministic 32 byte `seed`
- expectedResults are generated by `generate_expected_results_from_prompt()`

sigGen:

- supports negotiated internal/external interfaces
- supports deterministic and randomized groups
- internal `externalMu=false` tests contain `sk` and `message`
- internal `externalMu=true` tests contain `sk` and `mu`
- randomized tests contain deterministic `rnd`
- external pure tests contain `sk`, `message`, and `context`
- external preHash tests contain `sk`, `message`, `context`, and supported `hashAlg`
- in `nist-conformance`, ordinary groups have at least 15 tests; internal deterministic `externalMu=false` groups also include NIST rejection KAT inputs

sigVer:

- each generated group includes valid and invalid cases
- keypairs come from `keygen_internal()`
- valid signatures come from `siggen_internal()`
- invalid cases deterministically mutate message, mu, or signature-compatible input
- in `nist-conformance`, groups include 3 each of valid, modified message, modified signature commitment, modified signature z, and modified signature hint
- expectedResults are generated by the existing oracle path and include both `testPassed=true` and `testPassed=false`

Generated prompts are validated with `validate_mldsa_vector_set()`. Generated expectedResults are produced by `generate_expected_results_from_prompt()` and validated with `validate_mldsa_response()`.

## Phase 3-5 State Machine Interaction

Vector generation does not use wall-clock time for vector bytes. The Phase 3-5 clock is used only for lifecycle metadata:

- `createdAt`
- `updatedAt`
- `downloadedAt`
- `submittedAt`
- `validatedAt`
- `expiresAt`
- `stateHistory[*].at`

`GET /acvp/v1/vectorSets/{vectorSetId}` marks a `ready` vector set as `downloaded`. When every vector set in the session has been downloaded, the session becomes `vectorDownloaded`.

`POST /acvp/v1/vectorSets/{vectorSetId}/results` keeps Phase 3-4 matching-response behavior, but now records `resultsSubmitted`, `validating`, and `validated` or `failed` transitions.

## SHAKE

`SHAKE-128` and `SHAKE-256` may be accepted by the schema, but Phase 3-4 does not generate SHAKE preHash cases because the current API does not represent SHAKE output length. If another supported hash remains, SHAKE appears in warnings/unsupported entries and is excluded from generated groups. If only SHAKE remains for preHash generation, the request is rejected.

## Exclusions

The local skeleton still does not include:

- production database deployment
- JWT/login/mTLS
- production ACVP lifecycle
- vendor/module/OE/dependency resources
- frontend changes
- mldsa-native changes
- formal paging/query/error hardening
- async or large submission handling

See `backend/docs/acvp-v1-state-machine.md` for the supported local skeleton lifecycle. SQLite persistence is Phase 4-1, auth/JWT/mTLS remains Phase 4-2, paging/query/error hardening remains Phase 4-3, and async/large submission remains Phase 4-4.
