# ACVP Conformance Matrix

This document is based primarily on the official NIST ACVP Protocol Specification and the NIST ACVP ML-DSA JSON Specification. The current repository through Phase 4-3 Commit 3 is a local demo/prototype, not a production ACVP server. Phase 3-1 through Phase 3-5 add documentation, a formal namespace skeleton, local ML-DSA capabilities negotiation, deterministic local vector generation, and a local state machine. Phase 4-1 adds SQLite-backed local persistence. Phase 4-3 Commit 1 adds canonical nested vector set routes and ACVP array envelope responses for canonical `/acvp/v1` routes. Phase 4-3 Commit 2 maps local vector set validation into ACVP-style `results` objects and adds `showExpected`. Phase 4-3 Commit 3 adds local paging/query hardening and normalized ACVP error envelopes; it does not claim production-ready ACVP server conformance.

Official references:

- ACVP Protocol Specification: https://pages.nist.gov/ACVP/draft-fussell-acvp-spec.html
- ACVP ML-DSA JSON Specification: https://pages.nist.gov/ACVP/draft-celi-acvp-ml-dsa.html
- ACVP documentation landing page: https://pages.nist.gov/ACVP/
- FIPS 204: https://csrc.nist.gov/pubs/fips/204/final

Status values used in this matrix: `SUPPORTED`, `SUPPORTED-LOCAL`, `PARTIAL`, `MISSING`, `LOCAL_DEMO_ONLY`, `NOT_IN_SCOPE_YET`, `NEEDS_SPEC_REVIEW`.

The existing `/api/demo/acvp/...` routes are a local demo lifecycle and are not formal ACVP endpoints. The new `/acvp/v1/...` routes are skeleton endpoints under a formal namespace, not a production-ready server. Phase 3-4 vector generation, Phase 3-5 lifecycle state, and Phase 4-1 SQLite persistence are deterministic/local-fips204-skeleton behavior. Phase 4-3 moves canonical route local metadata under `extensions.localFips204Skeleton`, keeps `productionReady=false`, returns local ACVP-style vector set result dispositions, pages local list endpoints, and normalizes `/acvp/v1` errors. A formal production ACVP server still needs authentication/JWT/mTLS in Phase 4-2, async/large submission handling in Phase 4-4, production database deployment, security hardening, vendor/module/OE/dependency resources, and interoperability testing.

Phase 4-3 status summary:

- Earlier local milestones remain documented as Phase 3-2 namespace skeleton, Phase 3-3 capabilities negotiation, Phase 3-4 vector generation, Phase 3-5 state machine, and Phase 4-1 SQLite persistence.
- Phase 4-3A nested routes: implemented local.
- Phase 4-3B envelope: implemented local.
- Phase 4-3C results disposition: implemented local.
- Phase 4-3D paging/query: implemented local.
- Phase 4-3E error normalization: implemented local.
- Remaining protocol gaps move to Phase 4-2 auth/JWT/mTLS, Phase 4-4 async/large submission, and later vendor/module/OE/dependency resource phases.
