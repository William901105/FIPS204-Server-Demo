# ML-DSA Conformance Vector Generation

Phase 5-1 adds ML-DSA-specific conformance vector generation coverage for the gaps tracked as `CONF-005`, `CONF-006`, `CONF-008`, and `CONF-019` in `/root/ACVP204/conformance-results.md`.

This phase does not make the service a production ACVP server. `/acvp/v1` remains a local skeleton until auth/JWT, canonical result submission semantics, hidden non-sample expected results, async/large submission, and production deployment work are complete.

References:

- ACVP ML-DSA JSON Specification: https://pages.nist.gov/ACVP/draft-celi-acvp-ml-dsa.html
- ACVP Protocol Specification: https://pages.nist.gov/ACVP/draft-fussell-acvp-spec.html
- FIPS 204: https://csrc.nist.gov/pubs/fips/204/final
- ACVP-Server reference implementation under `/root/ACVP204/ACVP-Server`

## Profiles

Vector generation now accepts `generationProfile`:

- `local-debug`
- `nist-conformance`

If omitted, the default remains `local-debug` for backward compatibility.

`local-debug` keeps the previous fast skeleton behavior. `testsPerGroup` defaults to `2` and is capped at `10`.

`nist-conformance` applies NIST minimum counts per mode and is not capped by the local debug `MAX_TESTS_PER_GROUP=10`. If a request supplies a lower `testsPerGroup`, the generator automatically raises the effective count to the mode minimum:

| Mode | Minimum in `nist-conformance` |
| --- | ---: |
| keyGen | 25 |
| sigGen | 15 |
| sigVer | 15 |

Example:

```json
{
  "algorithms": [
    {
      "algorithm": "ML-DSA",
      "mode": "keyGen",
      "revision": "FIPS204",
      "parameterSets": ["ML-DSA-44", "ML-DSA-65", "ML-DSA-87"]
    }
  ],
  "campaignSeed": "00112233445566778899AABBCCDDEEFF00112233445566778899AABBCCDDEEFF",
  "testsPerGroup": 1,
  "generationProfile": "nist-conformance"
}
```

The generated `vectorGeneration` response records the requested `testsPerGroup`, the selected `generationProfile`, and `effectiveMinimums`.

## keyGen

Under `nist-conformance`, each generated keyGen group has at least 25 tests. Each test uses deterministic seed derivation and emits a 32-byte seed as 64 uppercase hex characters.

Expected results are still generated by `generate_expected_results_from_prompt()` through the native oracle. The generator does not hard-code `pk` or `sk`.

## sigGen

Under `nist-conformance`, each ordinary sigGen capability group has at least 15 tests.

The generator covers negotiated:

- `ML-DSA-44`, `ML-DSA-65`, `ML-DSA-87`
- `internal` and `external`
- `deterministic=true` and `deterministic=false`
- `externalMu=true` and `externalMu=false` for internal groups
- `pure` and `preHash` for external groups
- negotiated supported `hashAlg` values for preHash groups

Randomized sigGen tests use deterministic prompt-provided `rnd`. Deterministic groups do not include `rnd`.

## sigGen Rejection KAT

For `nist-conformance`, internal deterministic `externalMu=false` sigGen groups include the NIST draft rejection known-answer inputs:

- at least 5 rejection-outcome KAT cases from the `kats_table`
- at least 5 total-rejection-intended KAT cases from the `rejections_table`

The canonical ACVP prompt is not polluted with local metadata fields. KAT purpose is represented only in generator constants and tests.

Current limitation: Phase 5-1 verifies KAT inclusion and signature generation through the existing native oracle, but it does not instrument the native implementation to observe actual runtime rejection counts. Runtime rejection-count instrumentation remains a follow-up gap.

The ML-DSA-44 total-rejection table in the inspected draft contains two distinct rows. The generator cycles those NIST rows to provide five total-rejection-intended cases for the group while preserving deterministic KAT inputs. This is documented as an inclusion strategy, not as independent runtime count observation.

## sigVer

Under `nist-conformance`, each sigVer group has at least 15 tests with this distribution:

| Modification class | Count |
| --- | ---: |
| valid signature | 3 |
| modified message | 3 |
| modified signature commitment | 3 |
| modified signature z | 3 |
| modified signature hint | 3 |

Valid signatures are generated with `siggen_internal()`. Expected results are generated by `sigver_internal()`:

- valid signature -> `testPassed=true`
- all modification cases -> `testPassed=false`

Signature component offsets follow the packed signature layout from `mldsa-native/mldsa/src/params.h`: `c`, `z`, and `h`. The generator mutates a deterministic byte in the appropriate component.

## Out Of Scope

FIPS203 / ML-KEM backend work is out of scope for Phase 5-1. This phase does not add dummy FIPS203 providers, placeholder ML-KEM implementations, or frontend changes. FIPS203 will be integrated later through provider/interface work when the real backend lands.

Also out of scope:

- JWT/login/mTLS
- large submission / async validation
- vendor/module/OE/dependency resources
- changes to `/root/ACVP204/mldsa-native`
- canonical production ACVP result-submission behavior
